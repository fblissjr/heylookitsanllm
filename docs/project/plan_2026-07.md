## Direction -- why this plan exists

Sole-user hobbyist project. The grounding below was verified against primary
sources in July 2026 (inlined here so this stays self-contained):

- **MLX is healthy post-Awni.** Awni Hannun left Apple 2026-02-27 (now at
  Anthropic), but the core team stayed (Katharopoulos, Digani, Zhao, Collobert)
  and keeps cutting mlx-lm releases with a healthy community-PR flow -- MLX no
  longer depends on any individual.
- **The bet is baked into Apple's hardware + OS**, not a research project: M5
  Neural Accelerators are co-designed for MLX (~4x TTFT vs M4 -- though that gain
  does NOT reach the M2 Ultra this runs on); WWDC26 made MLX first-class in the
  Foundation Models framework (`MLXLanguageModel`); Ollama switched its
  Apple-Silicon engine off llama.cpp onto MLX (2026-03), NVIDIA contributing
  quantization + CUDA work.
- **No pivot to gguf/llama.cpp.** MLX wins steady-state decode (1.4-1.8x, up to
  3x on MoE) and memory (7-13%); llama.cpp still wins short-prompt prefill/TTFT,
  30K+ long-context decode, and architecture breadth -- none reverse the bet, and
  the radix prompt cache directly attacks MLX's one real weakness (repeated-prefix
  prefill). On the M2 Ultra the edge is memory architecture + decode speed, not
  the new silicon.
- **Verification-based decoding is a real wave, not flavor-of-the-month.** DFlash
  (z-lab, Feb 2026: a tiny block-diffusion drafter predicts ~16 tokens/forward,
  target verifies in one pass, lossless; native port `dflash-mlx` reports ~4x on
  Qwen3.5-9B at 89% acceptance) and DSpark (DeepSeek+PKU, Jun 2026: 60-85% faster,
  byte-identical, ships a draft-training repo) -- a ships-with-draft-head artifact
  is becoming as normal as "comes in 4-bit."
- **mlx-vlm is the stack's top bus-factor risk** (one maintainer) but alive and
  accelerating; LM Studio's unified mlx-engine independently formalized our
  pre-filled-cache split (mlx-lm always generates; mlx-vlm demoted to a vision
  add-on), Apple's official VLMs are Swift-only (mlx-swift-lm), and mlx-lm #993
  tracks Python multimodal.

Load-bearing conclusions:

- **The MLX bet holds; don't pivot.** Post-Awni MLX is healthier (M5
  co-design, WWDC-2026 Foundation Models, Ollama switched to MLX). Ride
  mlx-lm/mlx-vlm with minimal custom code.
- **Ecosystem posture (2026-07-06, full briefing:
  docs/architecture/ecosystem_strategy.md -- READ before perf
  or provider work):** the Python MLX stack is the right rail but is
  maintenance-mode infrastructure, not a living upstream (mlx-lm
  release-starved: 20+ unreleased commits, 259 open PRs, HF quietly
  co-maintaining; Apple's feature frontier ships in mlx-swift-lm).
  Operating rules: SHA-pin rather than wait for PyPI releases; check the
  open-PR backlog BEFORE writing any workaround; expect new capabilities
  (spec decoding especially) via sidecar packages like dflash-mlx, never
  via an mlx-lm release. Invariant to protect above all:
  `run_generation()` as the single chokepoint + the provider seam in
  config.py -- it converts ecosystem risk into a config entry (DFlash
  sidecar / Swift shim / llama-server all register as providers). The
  doc carries the tripwires (next mlx core wheel = load-bearing signal;
  day-one support location for the next frontier model = diagnostic) and

  then draft-model spec decode, then prefix-cache hit-rate, then
  memory-budget admission/batched decode; VLM pre-filled-cache path is
  incompatible with radix AND speculative prefill -- vision perf lives
  in the feature cache, not decode tricks.
- **Identity: an inference SCHEDULER for one machine with many demanding
  clients** (ComfyUI captioning, scripts, v3 frontend, agents) -- not an
  OpenAI-compatible endpoint (commodity). Differentiators: prefix cache,
  router/pinning/idle-unload, admission + batching, observability. Invest
  there.
- **Next capability wave: verification-based decoding** (DFlash/DSpark;
  lossless 3-4x decode). Our single chokepoint

   product reversal, not a bugfix.

6. **Deleting the batch endpoint closes spec §6's door** (a later

Fix shapes below are verified (Gemini triage claims were spot-checked
against source; their citations are unreliable, the verdicts are not).

1. **Streaming delivery quantization** -- `async_generator_with_abort`
   polls `chunk_future.done()` on `asyncio.sleep(0.1)`, capping SSE
   delivery AND recorded tok/s at ~10/s. Fix:
   `await asyncio.wait({chunk_future}, timeout=...)` (wakes on completion,
   keeps disconnect/keepalive cadence). Measure client-side tok/s
   before/after. Biggest UX win available AND the measurement prerequisite
   for everything in Phase 5.
2. **Honest headline metrics** (from the measurement audit): read native
   `chunk.prompt_tps`/`generation_tps` (mlx-lm computes them tightly
   around real prefill/decode; currently NEVER read anywhere in src/) as
   the recorded generation numbers; exclude queue_wait_ms from headline
   tok/s + TTFT (keep as its own field); success-only trend aggregation;
   fix /v1/messages prompt_tps formula (currently
   prompt_tokens / whole-request-elapsed). Small, mechanical,
   high-leverage.
3. **Executor quarantine list** -- dropping a close-timed-out executor
   lets GC fire the weakref callback -> shutdown sentinel -> wedged worker
   eventually exits -> the TLS teardown crash the pool exists to prevent.
   Keep a strong ref forever, never reuse. One line + test.
4. **Per-request reasoning parser** -- `_reasoning_parser` lives on the
   provider; an aborted stream leaves buffer state that pollutes the next
   request. Instantiate per request via `select_reasoning_parser`.
5. **Router TOCTOU + idle-unload race** -- `load_model()` runs outside
   `cache_lock` (~router.py:285); two different-model requests can both
   pass capacity; idle unload can tear down a provider between cache hit
   and generation start. Placeholder slot under lock; unload checks active
   count under the same lock. Rare solo, OOM-class consequences.
6. **Embedding pad-token guard** -- `padding=True` with no pad_token
   fallback (mlx_embedding_provider.py:199): `pad_token or eos_token` at
   load.
7. **mlx-vlm surface contract tests** (Direction strategy item 1) --
   tests/contract/test_mlxvlm_surface.py. An afternoon; precedent:
   test_snapshot_thread_affinity pins MLX stream semantics the same way.
8. **Import/re-import correctness** -- `already_configured` matches by id
   only (path never compared); re-import is skip-not-update. Match on
   resolved model_path too; add update-on-reimport (PUT semantics).
   (Correctness seed for the Phase 6 model-management redesign -- path
   identity is the prerequisite for a non-clobbering merge.)
9. **Upstream MLX bug -- SUPERSEDED 2026-07-07 by pre-submission
   verification: already fixed upstream, unreleased.** ml-explore/mlx#3619

## Phase 5 -- performance track (gated on FIXED telemetry)

### Measurement reality check (audited 2026-07-06 -- read before any perf work)

Owner asked whether our perf metrics are real or superficial/gameable.
Answer: mostly no, until Phase 1 items 1-2 land. Facts (verified):

- Streaming tok/s is recorded AFTER the 100ms poll boundary -- any model
  faster than ~10 tok/s records ~10 tok/s; a 60->48 regression is
  invisible. (This also caveats Swift tripwire 2 until fixed.) [Phase 1 item
  1 FIXED this in the server; the tests/e2e cadence guard (Phase 4 item 1,
  v1.34.8) now regression-tests DELIVERY from the client -- median inter-chunk
  gap < 50ms + > 30 tok/s on the MoE -- the one automated check that would
  catch a revert to poll-quantized delivery, since server telemetry cannot.]
- **Bit-identical output is NOT a valid gate for speculative decoding**
  (measured 2026-07-07, v1.34.12). Greedy spec-decode verifies draft tokens
  with a BATCHED forward whose float-accumulation order differs from sequential
  single-token decode, so borderline argmaxes flip -- fingerprints diverged on
  5/6 prompts while the run was still correct. optloop-lib's
  `require_fingerprint_match` therefore cannot certify a speculative (or any
  batched-verify / DFlash) run; those need a distributional/quality gate, not
  token identity. Doctrine, not a one-off: it applies to every
  verification-based-decoding experiment the perf track will run.
- Native mlx-lm prompt_tps/generation_tps: never read. /v1/messages
  prompt_tps: formula bug. Headline tok/s + TTFT include FIFO queue-wait
  (stamped before the gate acquires). Trends average failed/503 events at
  0.0 tok/s; means only, no percentiles; per-model bottlenecks computed
  but rendered by no shipped frontend. PerfCollector is in-memory
  (restart-losing); the durable JSONL streams have no reader (the
  metrics_dashboard.py that pyproject.toml's `rich` dep cites does not
  exist). RLM + batch traffic bypass telemetry entirely (record_request
  exists only in api.py/messages_api.py). model_load_ms is a >=100ms
  heuristic while the real measured load time sits unjoined in
  model_events.jsonl.
- optloop history: the retired app-level bench never touched the server
  path and never ran a cycle; greedy+fingerprint is a solid anti-drift
  guard (the "garbage but fast" hack is blocked) but self-referential --
  no ground-truth quality metric exists. docs/optimization_log.md is the
  real cross-session knowledge base (March findings incl. ceiling
  analysis: Python overhead <0.1% of per-token time; GPU forward pass
  dominates -- Gemma-3-27B bf16 at 79%, Qwen3.5-27B mxfp8 at 72% of
  bandwidth-bound max).

## Phase 6 -- model-management redesign (DIRECTION, not yet scoped)

New 2026-07-10 (owner-raised while scoping the j-space lens work). A captured
DIRECTION, not a build plan -- shape it into phased items in a later re-plan
pass. Coupled to the j-space work because lens management lands on the same
admin surface (see the j-space plan + `docs/frontend_v3.md`).

**Problem (grounded in the code, not vibes).** `models.toml` is brittle as an
interface:

1. **Scan regenerates, doesn't merge.** `model_importer.py` (525 lines) scans
   dirs + the HF cache and HAND-BUILDS toml text (`_model_to_toml_lines` ->
   `generate_toml` -> `f.write`), so a re-scan can clobber hand-edits.
   `admin_api.py` (453 lines) also writes the file.
2. **Identity is id-only.** `already_configured` matches by id, never resolved
   `model_path` (Phase 1 item 8) -- re-import is skip-not-update and path
   collisions aren't caught.
3. **No non-toml path to add/edit/enable a model.** It's hand-edit-the-toml or
   the CLI import -- both high-friction; hand-editing toml is the owner's
   stated pain point.

Note the file already bakes an absolute `model_path` per entry, so "point at a
model in any folder" is technically supported today -- the friction is in
GETTING an entry in/out, not in path flexibility.

**Direction.**

1. **Registry-over-scan: structured, non-clobbering.** Treat `models.toml` as
   generated-plus-overrides. A structured writer (tomlkit -- preserves
   comments/formatting) that MERGES by resolved `model_path`, never blind-
   regenerates. User overrides live in a layer a re-scan cannot stomp.
2. **"Point at a path, anywhere" as the primary add flow.** Add-by-path (file
   or dir, any location) -> sniff (reuse the `_is_mlx_model` /
   `_is_vision_model` / `_read_model_config` introspection) -> propose an entry
   -> confirm. No required folder convention.
3. **The v3 admin/model-management page is the real UI.** The `models` page
   already consumes `/v1/admin/models` + `/import` + `/scan` + load/unload;
   promote it from list+load/unload to full CRUD: add-by-path, edit
   id/tags/config, enable/disable, dedupe, re-scan-as-merge -- so the toml is
   an implementation detail, not the interface. Coupling recorded in
   `docs/frontend_v3.md`.
4. **Lens management lands on the same surface.** A fitted/converted j-space
   lens is another per-model artifact (`adapters/jspace/<model_id>/`); "fit a
   lens for this model" and "which models have a lens" belong next to the
   model CRUD. This coupling is what surfaced the redesign.

**Guardrails (do not regress).**

- `models.toml` still validates strictly (`extra="forbid"`, config.py) -- the
  structured writer emits only known fields; the invariant stays.
- Keep `model_path`-anywhere; never introduce a required directory layout.
- Sole-user hobbyist scope: stay lightweight. A structured merge + an admin
  CRUD page, NOT a model database or a package manager.

**Seeds already in the plan:** Phase 1 item 8 (import/re-import path identity +
PUT semantics) is the correctness prerequisite; Phase 2 item 5 (config_tui
retire) removes the drift-prone hand-editing helper this replaces.
